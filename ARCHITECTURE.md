# Архитектура сервиса

## Монолит vs Микросервисы

Для данного проекта выбран **монолитный подход**: один Flask-сервис, один Docker-образ.

**Обоснование:** сервис решает одну задачу — инференс модели. Дробить на микросервисы то, что логически едино, нет смысла. Преимущества монолита здесь: простота деплоя, нет сетевых задержек между компонентами, один контейнер = одна команда запуска.

Переход на микросервисы имел бы смысл при масштабировании: отдельный сервис инференса (горизонтально масштабируется), отдельный сервис препроцессинга, отдельный A/B-роутер на уровне API Gateway.

---

## Брокеры сообщений

В гипотетическом сценарии с высокой нагрузкой RabbitMQ использовался бы для:

- **Асинхронной пакетной обработки:** банк ночью пересчитывает риски для 500 000 клиентов. Записи публикуются в очередь, воркеры читают и делают инференс параллельно.
- **Логирования:** каждый API-запрос публикует событие в отдельный топик, потребители (аналитика, мониторинг, аудит) обрабатывают независимо.

---

## Логирование

Каждый запрос и ответ логируются в stdout в JSON-формате. В production логи собирались бы ELK-стеком (Elasticsearch + Logstash + Kibana) для real-time мониторинга.

---

## ONNX

Для оптимизации инференса модель можно конвертировать в ONNX через `skl2onnx`. Инференс через `onnxruntime` быстрее sklearn в 2–5x и не требует scikit-learn в production-контейнере.

---

## uWSGI + NGINX

Встроенный Flask-сервер однопоточный и не подходит для production. В реальной системе: NGINX как reverse-proxy (SSL, rate limiting, балансировка) → uWSGI с несколькими Python-воркерами → Flask. Для данного проекта Flask запускается напрямую, что достаточно для демонстрации.

---

## DVC и MLflow

**DVC** — версионирование данных и моделей (аналог Git для бинарных файлов). Позволяет воспроизвести любую версию пайплайна: `dvc repro` пересчитывает всё от данных до модели.

**MLflow** — отслеживание экспериментов. Логирует параметры, метрики и артефакты, позволяет сравнивать запуски через UI и выбирать лучшую модель для деплоя.
